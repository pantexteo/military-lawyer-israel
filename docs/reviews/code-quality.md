## Medium issues

### 1. `btn.firstChild.textContent` — хрупкая манипуляция DOM

`index.html:3201`

```js
btn.firstChild.textContent = expanded ? 'Свернуть ' : 'Ещё ' + hidden + ' вопросов ';
```

Разметка кнопки сейчас:
```html
<button class="faq__more-btn" id="show-more-faqs" aria-expanded="false">
  Ещё 10 вопросов
  <svg ...>...</svg>
</button>
```

`firstChild` здесь — текстовый узел благодаря тому, что перед текстом «Ещё 10 вопросов» стоит перевод строки + пробелы. Любое форматирование (например, если кто-то оборачивает SVG или переносит текст, или меняет отступы) может привести к тому, что `firstChild` станет самим SVG или пустым текстовым узлом, и кнопка перестанет менять подпись. Более надёжный паттерн — обернуть лейбл в `<span class="faq__more-btn-label">` и обновлять его напрямую.

Плюс: при `hidden === 1/2/3/4` текст «вопросов» будет грамматически некорректным (нужно «вопрос»/«вопроса»). На сегодня `hidden` захардкожен в 10, так что это теоретическая проблема, но легко снимается вынесением строки в функцию-плюрализатор или просто переводом на нейтральную формулировку.

Рекомендация: вынести label в отдельный span + вынести строку сборки текста в отдельную переменную/функцию.

### 2. Дублирование значения «10» в разметке и в JS

`index.html:2825` статически содержит `Ещё 10 вопросов`, а JS на строке 3201 пересчитывает число из `list.querySelectorAll('.hidden-faq').length`. Если кто-то добавит/удалит `hidden-faq` и забудет поправить HTML-лейбл — пользователь после первого клика «Свернуть → снова открыть» увидит правильное число, но при первой загрузке — устаревшее. Исправление: рассчитывать начальный лейбл тоже из JS (один источник истины).

### 3. Жёстко захардкоженные селекторы CTA-ссылок в двух местах

`index.html:3223` и `scripts/inject_cta_tracking.py:36`

```js
e.target.closest('a[href*="t.me/juli_x11"], a[href*="wa.me/79139406920"]')
```

Telegram username и WhatsApp-номер захардкожены одновременно:
- в CLAUDE.md упомянут плейсхолдер `@YOUR_TELEGRAM` для замены перед деплоем;
- `juli_x11` и `79139406920` уже раскиданы по HTML (это нормально, их там и должно быть — как href-атрибуты);
- но трекер отдельно их хранит в двух местах (index.html и Python-скрипт).

Если контакт изменится — редактируется только HTML-href, и трекер тихо перестанет срабатывать (селектор не найдёт совпадений, событий `cta_click` в GA4 не будет). Предложение: хранить список в data-атрибуте на элементе (`data-cta-type="telegram|whatsapp"`) и селектор трекера делать на `a[data-cta-type]`. Тогда при обновлении контакта нужно только поправить href, а трекер продолжит работать.

Это не срочная проблема, но стоит внести в беклог — особенно с учётом того, что плейсхолдер `@YOUR_TELEGRAM` в CLAUDE.md прямо говорит о будущей замене.

## Nits

### 1. Magic values без комментариев в смещениях scroll

Существующий код (не из этого PR) на строке 3160 содержит `var offset = 44 + 64 + 20; // emergency bar + nav + gap`. Новые блоки этот паттерн соблюдают — ок.

### 2. `for`-loop вместо `.find()` в трекере

`index.html:3228` и `inject_cta_tracking.py:41`:
```js
for (var i = 0; i < LOCATIONS.length; i++) {
  if (a.closest(LOCATIONS[i][0])) { button_location = LOCATIONS[i][1]; break; }
}
```

Стилистически — в файле уже используется `var` и `for`-loop (ES5-стиль для совместимости), так что это согласуется с кодом. `LOCATIONS.find(...)` был бы чище, но не критично.

### 3. `rel` в Python: `if rel == Path("index.html")`

`scripts/inject_cta_tracking.py:77`

Сравнение путей через `Path("index.html")` работает, но `rel.name == "index.html" and rel.parent == Path(".")` читается явнее. Мелочь.

### 4. Комментарий `// cta_click_tracker_v1` в двух местах

Маркер версии хорош для идемпотентности (используется как `TRACKER_MARKER`), но в самом `index.html` (строка 3205) его нет — там комментарий `// ---- GA4: track CTA clicks (Telegram + WhatsApp) ----`. Если однажды понадобится синхронно обновить трекер во всех файлах, полезно было бы оставить и в index.html маркер `cta_click_tracker_v1` как единый идентификатор версии.

### 5. `try/catch` вокруг `gtag` в production

`index.html:3231-3237`:
```js
try {
  gtag('event', 'cta_click', { ... });
} catch (err) { /* noop */ }
```

Защита разумная (на случай AdBlocker, CSP-блокировок), но комментарий `/* noop */` не объясняет, почему глотаем ошибку. Одна строка комментария (`// gtag может падать при AdBlock/CSP — молча игнорим`) помогла бы будущему себе.

## Что хорошо сделано

- **Идемпотентность Python-скрипта**: три независимые проверки (`GA_ID_MARKER`, `TRACKER_MARKER`, `</body>`) и чёткие возвращаемые статусы — легко понять, что произошло с каждым файлом.
- **Чистое разделение concerns**: FAQ show-more и GA4-трекер — два независимых IIFE, никак не связанные; легко удалить или заменить по отдельности.
- **Консистентность стиля**: новый CSS и JS следуют существующим конвенциям файла (BEM, `var`, IIFE, `{ passive: true }` там где нужно).
- **Accessibility**: новая кнопка «Ещё 10 вопросов» имеет `aria-expanded`, оно правильно обновляется в JS. CSS-правило `.faq__list.expanded + .faq__more .faq__more-btn svg { transform: rotate(180deg); }` — изящный способ поворота стрелки без JS.
- **Делегирование событий в трекере**: один `addEventListener` на `document` вместо навешивания на каждую ссылку — корректно и производительно, плюс работает для динамически добавленных элементов.
- **Корректная вставка в dezertir-cahala.html**: блок пришит перед `</body>`, отступы и структура сохранены.
- **Guard clauses в начале каждого IIFE**: `if (!btn || !list) return;` и `if (typeof gtag !== 'function') return;` — предохраняют от падений на страницах, где этих элементов нет.
